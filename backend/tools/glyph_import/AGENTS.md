# AGENTS.md

## Назначение

Этот репозиторий содержит вспомогательный offline-инструмент для подготовки glyph dataset из фото и сканов бумажных схем букв для филейного вязания.

Инструмент не является частью production backend API.

Его задача:

* принимать изображения схем букв
* выполнять предобработку изображения
* извлекать клеточную структуру
* определять заполненность клеток
* строить матрицу `0/1`
* формировать черновик glyph JSON
* сохранять результат в файл

---

## Границы ответственности

Инструмент отвечает только за подготовку данных для библиотеки глифов.

Он НЕ должен:

* реализовывать API основного backend
* изменять production-логику генерации схем
* добавлять HTTP endpoints без явной необходимости
* смешивать runtime-логику backend и tooling-логику импорта
* экспортировать PDF или изображения как пользовательскую функцию основного сервиса

---

## Контекст проекта

Этот инструмент находится внутри основного backend-репозитория, но должен рассматриваться как отдельная подсистема.

Основной backend:

* читает готовый glyph dataset
* использует его для генерации схем из текста

Этот инструмент:

* подготавливает glyph dataset
* сохраняет промежуточные и итоговые артефакты
* может использовать вспомогательные debug-выводы и preview

---

## Основная цель версии 1

Нужно реализовать локальный pipeline для сценария:

* на вход подается фото или скан бумажной схемы одной буквы или символа
* система пытается извлечь клеточную матрицу
* система формирует черновой glyph JSON
* результат можно проверить вручную

На первом этапе важнее:

* прозрачность
* предсказуемость
* простота отладки

чем “умность” или автоматическая универсальность.

---

## Технический подход

Для первой версии использовать в первую очередь классические алгоритмы обработки изображений, а не AI/ML.

Разрешено использовать:

* Python
* OpenCV
* Pillow
* numpy
* pydantic
* pytest

Предпочтительный подход:

1. загрузка изображения
2. приведение к grayscale
3. бинаризация
4. при необходимости выравнивание перспективы
5. поиск сетки
6. разбиение на клетки
7. классификация клетки как пустой или заполненной
8. сборка матрицы
9. экспорт glyph JSON
10. сохранение preview/debug артефактов

---

## Использование AI/ML

AI/ML не является основным подходом для версии 1.

Не нужно сразу:

* добавлять нейросети
* обучать модели
* строить сложный ML pipeline
* использовать OCR как основную стратегию

AI/ML можно рассматривать только если станет ясно, что:

* классические алгоритмы не справляются
* изображения слишком шумные
* сетка не детектируется надежно
* ручная корректировка слишком велика

До этого момента приоритет у classical CV.

## Архитектурные правила

Нужно соблюдать разделение по слоям:

* `domain` — модели данных и контракты
* `services` — отдельные шаги обработки
* `pipelines` — orchestration
* `utils` — мелкие вспомогательные функции

Не писать всю логику в одном файле.

Не смешивать:

* загрузку файлов
* обработку изображения
* построение матрицы
* экспорт JSON

в один монолитный модуль.

---

## Правила структуры

Предпочтительная структура:

* `src/main.py` — entrypoint
* `src/config.py` — конфигурация
* `src/domain` — модели
* `src/services` — обработка
* `src/pipelines` — pipeline
* `src/utils` — вспомогательные функции
* `input/raw` — исходные изображения со схемами
* `input/extracted` — вырезанные отдельные символы или папки с извлечёнными символами
* `input/debug` — промежуточные и отладочные изображения, связанные с extraction
* `output/glyphs` — итоговые glyph JSON
* `output/manifests` — manifest.json и другие файлы-манифесты
* `tests` — тесты

## Расширение сценария инструмента

Помимо сценария «одна картинка = один символ», инструмент также поддерживает подготовительный этап extraction:

* на вход может подаваться jpg/png с несколькими символами
* инструмент может извлекать отдельные символы в папку `input/extracted`
* инструмент может сохранять промежуточные debug-артефакты в `input/debug`
* результат extraction должен сопровождаться манифестом в `output/manifests`

Extraction является частью `glyph_import` и не должен выноситься в отдельный tool без явной необходимости.

* Использовать type hints

* Не добавлять web API без явной задачи

## Формат результата

Основной результат работы инструмента:

* матрица `0/1`
* char, если он задан пользователем
* width
* height
* JSON-структура, пригодная для включения в glyph dataset

Инструмент может дополнительно сохранять:

* preview изображения
* debug изображения
* промежуточные шаги обработки

## Правила изменений

Перед добавлением нового крупного шага:

* проверить, не нарушает ли он scope инструмента
* не переносит ли tooling-логику в production backend
* не добавляет ли избыточный ML/AI раньше времени

Если задача неоднозначна, предпочтение отдавать:

* простым решениям
* прозрачным алгоритмам
* промежуточным debug-артефактам
* удобству ручной проверки

Если в структуре `input/` или `output/` уже есть пользовательские файлы, их нельзя удалять или переименовывать без необходимости.

Если назначение существующего файла неоднозначно, инструмент не должен угадывать:
* такой файл нужно оставить на месте или переместить в наиболее безопасное место
* о таком файле нужно явно сообщить в итоговом отчёте

## Экономия контекста

Для задач внутри `tools/glyph_import` не читать документацию основного backend
(`PRODUCT_REQUIREMENTS.md`, `DOMAIN.md`, `GENERATION_RULES.md`,
`API_CONTRACT.md`, `ARCHITECTURE.md`, корневой `TASKS.md`), если задача явно
касается только offline-инструмента extraction/import.

Для маленьких правок сначала читать только:

* `tools/glyph_import/AGENTS.md`
* `tools/glyph_import/TASKS.md`
* конкретные файлы кода/тестов, связанные с задачей

Не открывать большие файлы целиком, если достаточно `rg` и фрагмента через
`Select-Object -Skip/-First`.

Для визуальной проверки не просматривать все изображения подряд. Сначала
проверять только файлы, явно названные в задаче. Расширять список только если
найдена общая причина.

Не запускать полный `pytest` после каждой маленькой правки. Сначала запускать
релевантные unit-тесты, полный набор тестов — перед финальным ответом или после
изменения общего pipeline.
