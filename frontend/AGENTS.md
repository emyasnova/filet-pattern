# AGENTS.md

## Роль Codex

Ты работаешь над frontend-приложением для генерации схем филейного вязания.

Приложение должно быть реализовано как публичное web-приложение на:

- React
- Vite
- TypeScript

Backend в MVP не используется. Telegram-бот и Telegram Mini App не входят в MVP, но архитектура не должна мешать их добавлению в будущем.

## Главная цель MVP

Создать редактор схем филейного вязания, где пользователь может:

1. Видеть рабочую область 100 × 100 клеток.
2. Изменять размер рабочей области.
3. Видеть справа список мотивов, загруженных из JSON-файлов.
4. Поворачивать и отражать каждый мотив в правой панели.
5. Перетаскивать мотивы в рабочую область.
6. Видеть preview места вставки при drag-and-drop.
7. Заполнять и очищать клетки рабочей области кликом.
8. Экспортировать рабочую область в PNG.
9. Сохранять рабочую область в JSON.
10. Загружать ранее сохраненную рабочую область из JSON.
11. Очищать рабочую область.

## Важные продуктовые правила

- `1` — заполненная клетка.
- `0` — пустая видимая клетка.
- `null` — невидимая/прозрачная клетка в мотиве.
- При отображении мотива `null` выглядит как пустая клетка.
- При переносе мотива на рабочую область `null` не изменяет клетку рабочей области.
- При переносе мотива за границы рабочей области лишняя часть обрезается.
- Каждая карточка мотива справа хранит свое текущее transformed-состояние.
- Именно transformed-состояние карточки переносится в рабочую область.
- Рабочая область хранит только `0` и `1`.

## Правила изменения размера рабочей области

- Размер по умолчанию: 100 × 100 клеток.
- Пользователь может изменить ширину и высоту.
- При увеличении размера существующее содержимое сохраняется.
- Новые клетки добавляются как пустые `0`.
- При уменьшении размера содержимое за новыми границами обрезается.

## Правила PNG-экспорта

PNG должен содержать:

- белый фон;
- черные заполненные клетки;
- тонкую серую сетку;
- более яркие разделители блоков по 10 клеток;
- текст с шириной и высотой схемы в клетках;
- без лишних подписей и декоративных элементов.

## Архитектурные принципы

Старайся отделять бизнес-логику от React-компонентов.

Желательная структура:

```text
src/
  domain/
  application/
  infrastructure/
  ui/
```

Где:

- `domain` — типы, структуры данных, чистые операции над матрицами;
- `application` — сценарии использования: вставить мотив, изменить размер, сохранить, загрузить;
- `infrastructure` — загрузка JSON, экспорт PNG, работа с файлами;
- `ui` — React-компоненты и хуки.

## Стиль работы

Перед крупными изменениями:

1. Кратко опиши план.
2. Укажи, какие файлы будут изменены.
3. После реализации перечисли, что сделано.
4. Укажи, как проверить результат вручную.

Не добавляй backend, авторизацию, базу данных, Telegram-интеграцию, платные функции или сложный дизайн без отдельной задачи.

## Качество кода

- Используй TypeScript-типы для всех ключевых структур.
- Не храни бизнес-логику прямо в JSX.
- Не дублируй операции поворота/отражения в разных компонентах.
- Не мутируй матрицы напрямую без необходимости.
- Старайся писать функции так, чтобы их можно было тестировать отдельно.
- Обрабатывай ошибки загрузки JSON и импорта пользовательского файла.

## Проверка

После изменений по возможности запускай:

```bash
npm run build
npm run lint
```

Если в проекте добавлены тесты:

```bash
npm test
```

Если команды отсутствуют, не выдумывай результат их выполнения. Сообщи, что команды пока не настроены.

## Экономный режим работы Codex

- Не запускай автоматически `npm run build`, `npm run lint` и `npm test`.
- Запускай эти команды только если пользователь явно попросил или если без них нельзя безопасно завершить задачу.
- Для чтения файлов используй точечные `rg`, `Select-String`, `Get-Content` с `-TotalCount` / `Select-Object -First/-Skip`, а не чтение всего файла целиком.
- Для следующих фаз пиши короче: краткий план, список файлов, краткий итог без подробного пересказа.
- Если проверка не запускалась из-за этого правила, прямо укажи это в итоговом сообщении.
