# PROJECT_SPEC.md

## Назначение приложения

Приложение предназначено для создания схем филейного вязания. Пользователь собирает итоговую схему на клеточной рабочей области из готовых JSON-мотивов и ручного редактирования клеток.

## Целевая платформа

MVP — публичное web-приложение.

Не входят в MVP:

- backend;
- авторизация;
- база данных;
- Telegram-бот;
- Telegram Mini App;
- платные функции;
- совместное редактирование.

## Основные сущности

### Cell

Клетка мотива может иметь значения:

```ts
type PatternCell = 0 | 1 | null;
```

Клетка рабочей области может иметь значения:

```ts
type CanvasCell = 0 | 1;
```

Правила:

- `0` — пустая клетка;
- `1` — заполненная клетка;
- `null` — прозрачная клетка мотива.

`null` используется только в мотивах. Рабочая область хранит только `0` и `1`.

### Pattern

Готовый мотив из JSON:

```ts
interface Pattern {
  char?: string;
  name?: string;
  width: number;
  height: number;
  cells: PatternCell[][];
}
```

Пример структуры:

```json
{
  "char": "B",
  "width": 18,
  "height": 21,
  "cells": [
    [0, 0, 1],
    [0, 1, 0]
  ]
}
```

Требования к валидации:

- `width` должен быть положительным числом;
- `height` должен быть положительным числом;
- `cells.length` должен соответствовать `height`;
- каждая строка `cells` должна соответствовать `width`;
- значения клеток могут быть только `0`, `1`, `null`.

### Canvas

Рабочая область:

```ts
interface CanvasState {
  width: number;
  height: number;
  cells: CanvasCell[][];
}
```

Размер по умолчанию:

```text
100 × 100 клеток
```

## Функциональные требования

### FR-01. Отображение рабочей области

Пользователь видит слева рабочую область в виде таблицы квадратных клеток.

Требования:

- по умолчанию размер 100 × 100;
- клетки квадратные;
- пустые клетки белые;
- заполненные клетки черные;
- сетка видима;
- каждые 10 клеток визуально отделяются более заметной линией.

### FR-02. Редактирование клетки кликом

При клике по клетке рабочей области:

- если клетка пустая `0`, она становится заполненной `1`;
- если клетка заполненная `1`, она становится пустой `0`.

### FR-03. Изменение размера рабочей области

Пользователь может изменить ширину и высоту рабочей области.

Правила:

- при увеличении размера существующие клетки сохраняются;
- новые клетки заполняются `0`;
- при уменьшении размера содержимое за новыми границами обрезается.

### FR-04. Загрузка мотивов из JSON

Приложение загружает список мотивов из:

```text
public/patterns/index.json
```

Затем загружает каждый файл из списка.

Если отдельный мотив не загрузился или не прошел валидацию, приложение не должно падать целиком. Нужно показать ошибку для этого мотива или пропустить его с сообщением в UI/console.

### FR-05. Правая панель мотивов

Справа отображается вертикальный список карточек мотивов.

Требования:

- мотивы расположены друг под другом;
- у правой области есть вертикальный скролл;
- каждая карточка показывает текущую версию мотива;
- каждая карточка имеет кнопки:
  - повернуть на 90°;
  - отразить по горизонтали;
  - отразить по вертикали.

### FR-06. Трансформации мотивов

Каждая карточка мотива хранит собственное текущее transformed-состояние.

Операции:

- rotate 90° clockwise;
- flip horizontal;
- flip vertical.

После трансформации должны корректно обновляться:

- `width`;
- `height`;
- `cells`.

### FR-07. Drag-and-drop мотива

Пользователь может перетащить мотив из правой панели на рабочую область.

Правила:

- переносится текущее transformed-состояние карточки;
- позиция вставки определяется по клетке рабочей области, над которой происходит drop;
- верхний левый угол мотива вставляется в выбранную клетку;
- если часть мотива выходит за границы рабочей области, лишняя часть обрезается;
- `1` из мотива записывает `1` в рабочую область;
- `0` из мотива записывает `0` в рабочую область;
- `null` из мотива ничего не меняет в рабочей области.

### FR-08. Preview вставки

Во время drag-over рабочей области пользователь видит preview области, куда будет вставлен мотив.

Требования:

- preview должен соответствовать текущему transformed-состоянию мотива;
- preview должен обрезаться по границам рабочей области;
- желательно визуально отличать клетки, которые будут заполнены, очищены или не изменятся;
- для MVP достаточно показать полупрозрачный overlay в зоне вставки.

### FR-09. Очистка рабочей области

Кнопка “Очистить” сбрасывает все клетки рабочей области в `0`.

Перед очисткой желательно запросить подтверждение, чтобы пользователь случайно не потерял работу.

### FR-10. Сохранение рабочей области в JSON

Кнопка “Сохранить JSON” скачивает файл текущей рабочей области.

Формат:

```json
{
  "version": 1,
  "type": "filet-crochet-canvas",
  "width": 100,
  "height": 100,
  "cells": [
    [0, 1, 0],
    [1, 0, 1]
  ]
}
```

### FR-11. Загрузка рабочей области из JSON

Пользователь может выбрать ранее сохраненный JSON-файл.

После загрузки:

- размер рабочей области берется из файла;
- клетки заменяются содержимым файла;
- файл валидируется перед применением;
- при ошибке валидации пользователь видит понятное сообщение.

### FR-12. Экспорт PNG

Кнопка “Сгенерировать изображение” скачивает PNG.

PNG содержит:

- белый фон;
- черные заполненные клетки;
- тонкую серую сетку;
- более яркие разделители блоков по 10 клеток;
- текст с шириной и высотой схемы в клетках;
- без лишних подписей.

## Нефункциональные требования

- Приложение должно работать без backend.
- Приложение должно быть пригодно для деплоя как статический сайт.
- Бизнес-логика должна быть отделена от UI.
- Основные операции над матрицами должны быть чистыми функциями.
- Код должен быть читаемым и типизированным.
- UI должен оставаться usable при большом количестве мотивов.
